# 0001. Codex automation layer is Rust-owned extraction, not vendor harness

- Status: accepted
- Date: 2026-06-23

## Context

The repository tracks a large `.claude` automation surface: agents, commands, hooks, helpers, skills, settings, and Rusty IDD adapter files. A raw `.codex` mirror alone creates skill overload, MCP/config rot, token burn, and ambiguous team routing. Prior attempts treated `.codex` as prompt/config parity or as material for a vendor harness. That is the wrong ownership layer.

## Decision

`.claude` is source material. `.codex` is the Rust automation extraction frontier generated by `crates/codex-env`. The generator must preserve raw source evidence while compiling lower-token, crate-owned automation metadata such as teams, hook routes, command/prompt routes, MCP declarations, and Rusty IDD integration. Vendor harnesses are not the destination for this automation.

## Consequences

- `crates/codex-env` owns deterministic extraction and validation.
- `.codex/mirror/.claude` remains byte-for-byte source evidence, not the primary runtime representation.
- Generated runtime surfaces can be normalized for Codex and current checkout paths without editing `.claude`.
- Tests and doctor checks must catch empty teams, prompt-global pollution, stale absolute runtime paths, and mirror drift.
