> Evidence-based interpretation and explanation generation for bioacoustic AI.

> **⚠️ Status: work-in-progress skeleton.** This crate does not yet expose a
> functional public API. The crate root publishes only the `VERSION` constant.
> Source files for the planned design exist in the tree but are **not wired into
> `lib.rs`**, so they are not reachable from `sevensense_interpretation::`. The
> sections below describe the *intended* design, not currently-callable code.

**sevensense-interpretation** is intended to generate human-readable, evidence-backed
explanations for AI predictions. The planned design centers on "evidence packs" that
document why a species was identified, what context contributed to the decision, and
how confident the system is — useful for scientific credibility and audit trails.

## Current Public API

The only item currently exported at the crate root is:

```rust
use sevensense_interpretation::VERSION;

fn main() {
    println!("sevensense-interpretation v{VERSION}");
}
```

| Item | Kind | Description |
|------|------|-------------|
| `VERSION` | `pub const &str` | Crate version (`CARGO_PKG_VERSION`) |

No other types, traits, or functions are part of the compiled public API yet.

## Installation

Add to your `Cargo.toml`:

```toml
[dependencies]
sevensense-interpretation = "0.1"
```

## Planned / Not Yet Implemented

The following describes the intended bounded context. The corresponding source
files (`domain/`, `application/`, `infrastructure/`, `templates.rs`) exist but are
**not** declared as modules in `lib.rs`, so none of these types are accessible
today. **Do not depend on them.**

### Intended structure

```text
sevensense-interpretation (planned)
├── domain/
│   ├── entities      # EvidencePack, EmbeddingId, Interpretation, Claim, ...
│   └── repository    # EvidencePackRepository, ClusterRepository
├── application/
│   └── services      # InterpretationService
├── infrastructure/
│   ├── evidence_builder  # EvidenceBuilder
│   └── claim_generator   # ClaimGenerator, ClaimBuilder
└── templates         # InterpretationTemplates, EvidencePackFormatter
```

### Planned domain entities

The (currently unwired) domain layer defines, among others:

- `EvidencePack` — a query embedding id plus neighbor evidence, cluster context,
  optional sequence context, and a generated `Interpretation`. Constructed via
  `EvidencePack::new(query_embedding_id, neighbors, cluster_context, sequence_context, interpretation)`.
- `EmbeddingId` — newtype identifier (`EmbeddingId::new("...")`, `::generate()`).
- `Interpretation`, `Claim`, `EvidenceRef`, `EvidenceRefType` — structural
  description plus evidence-cited claims.
- `NeighborEvidence`, `ClusterContext`, `SequenceContext` — supporting context.

### Planned repository trait

```rust,ignore
// Planned — not yet part of the public API.
use sevensense_interpretation::domain::repository::EvidencePackRepository;

#[async_trait::async_trait]
pub trait EvidencePackRepository: Send + Sync {
    async fn save(&self, pack: &EvidencePack) -> Result<()>;
    async fn find_by_id(&self, id: &str) -> Result<Option<EvidencePack>>;
    async fn find_by_query(&self, embedding_id: &EmbeddingId) -> Result<Vec<EvidencePack>>;
    // ...
}
```

An `InMemoryEvidencePackRepository` is provided in the source for testing.

### Planned application service

`InterpretationService` is intended to build an `EvidencePack` for a query
embedding (`build_evidence_pack(&query_id)`), pulling neighbors from a
`VectorSpaceService` and cluster context from a `ClusterRepository`.

### Planned formatting

`InterpretationTemplates` and `EvidencePackFormatter` are intended to render an
`EvidencePack` to a human-readable report, a compact summary, or JSON.

> Note: there is currently **no LLM / provider integration** in this crate (no
> Claude, GPT, or local-model code). The planned interpretation is
> template-driven over structured evidence, not generated by a language model.
